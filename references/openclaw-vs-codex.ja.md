# OpenClaw vs Codex Skill Creator

> バンドルされたCodexスキルクリエイター（`/app/skills/skill-creator/`）と、新しいOpenClawネイティブスキルクリエイターの違いを文書化します。新しいOpenClawスキルの作成や、既存のCodexスキルの移植時にこのリファレンスを使用してください。

## アーキテクチャの違い

| | Codex (CLI コーディングエージェント) | OpenClaw (マルチエージェント) |
|---|---|---|
| 実行 | 単一CLIセッション | メイン + sub-agent (spawn/fork) |
| コンテキスト | すべてのコンテキストがセッション内に存在 | sub-agentは**空の**コンテキストで開始 |
| スキル読み込み | description → body → scripts (順次) | description(常時) → body(トリガー時) → references(オンデマンド) |
| スキル保存先 | 単一の場所 | workspace(再起動間で維持) vs bundled(アップデート時に初期化) |
| プロンプト注入 | 不要 (単一コンテキスト) | **必須** (sub-agentへの明示的注入) |
| パッケージング | `package_skill.py`が`.skill`ファイルを作成 | パッケージング不要 — フォルダ構造を直接認識 |

## 5つの主要な違い

### 1. Sub-Agent コンテキスト注入

**Codex**: すべての作業が1つのセッションで行われるため、コンテキスト共有が自然です。

**OpenClaw**: sub-agentをspawnすると、コンテキストが**空**になります。メインエージェントがPhase 0で収集した情報をsub-agentのプロンプトに明示的に注入する必要があります。

Codexスキルにはこの概念がありません。OpenClawでは最も重要なパターンです。

```
Codex:   "このテキストを要約して" → OK (コンテキストが既に存在)
OpenClaw: "このテキストを要約して" → FAIL (sub-agentにコンテキストがない)

OpenClawの正しいアプローチ:
"このテキストは「OOOユーザーマニュアル」のセクション1〜5です。
ドキュメント全体は5パート27セクションです。このチャンクは「新しい設計手法」を扱っています。
以下を保持してください：[比喩]、[引用]、[ケーススタディ]..."
```

### 2. Style Guide 優先

**Codex**: "簡潔さが鍵" — コンテキストウィンドウの節約に集中します。エージェントが既に十分に賢いと仮定します。

**OpenClaw**: Style Guideがないと、sub-agentは「テキスト圧縮機」になります。**判断基準**が必須です。

Codexスキルは形式（構造、長さ）に集中しますが、OpenClawスキルは判断基準（何を保持するか、どのトーンで、どの視点で）も含める必要があります。

### 3. Phase 0 (分析段階) が必須

**Codex**: エージェントが作業しながら自然に分析します。

**OpenClaw**: メインエージェントがPhase 0で分析し、結果をsub-agentプロンプトに注入するパイプラインが必要です。

Phase 0がないと：
- 各チャンクが独立して処理されます（全体の流れが失われます）
- チャンク間で重複コンテンツが現れます
- 著者の声（比喩、引用）が欠落します

### 4. 順次パイプライン

**Codex**: 複数の出力を並列で生成でき、エージェントが自然にマージします。

**OpenClaw**: 複雑な出力は、単純から複雑へ順次拡張する必要があります。

例：要約の深さ 1x + 5x
- Codex: 並列生成可能（エージェントがマージ）
- OpenClaw: 1x → 5x 順次（1xの構造が5xの骨組みとして機能）

### 5. パッケージング不要

**Codex**: `package_skill.py`が配布用の`.skill`ファイル（実質的にzip）を作成します。

**OpenClaw**: フォルダ構造を直接認識します。スキルが機能するために`SKILL.md`だけで十分です。

パッケージングスクリプト、検証スクリプト、initスクリプトは不要です。

## 移植チェックリスト：Codexスキル → OpenClawスキル

CodexスキルをOpenClawに移植する際：

- [ ] **sub-agentが必要か確認** — スキルがチャンキングや並列処理を行う場合、Phase 0 + コンテキスト注入パターンが存在するか確認
- [ ] **Style Guideはあるか？** — なければ追加してください。形式仕様だけではsub-agentが高品質な出力を生成できません
- [ ] **Phase 0を追加** — 作業開始前に分析段階があるか？
- [ ] **`[ ]`プレースホルダー付きプロンプトテンプレート** — プレースホルダーがPhase 0の結果で埋められているか確認
- [ ] **マージ指示** — マルチチャンク出力の単純な連結を明示的に防止
- [ ] **`package_skill.py`参照の削除** — OpenClawでは不要
- [ ] **`init_skill.py`参照の削除** — OpenClawでは不要
- [ ] **descriptionにトリガーキーワード** — 英語とローカライズされたキーワードの両方を含める
- [ ] **「使用タイミング」をbodyからdescriptionへ移動** — OpenClawはトリガー後にのみbodyを読み込みます

## Codexから維持すべきパターン

以下は両環境でうまく機能する良いパターンです：

1. **Progressive Disclosure** — SKILL.mdを500行以下に維持し、詳細はreferences/に移動
2. **"簡潔さが鍵"原則** — エージェントが既に知っていることを再説明しない
3. **自由度の調整** — タスクの脆弱性（高/中/低）に応じて具体性を調整
4. **単一レベル参照深度** — references/内から他のreferences/を参照しない
5. **使用後の改善** — スキルを使用し、失敗を観察し、修正し、繰り返す
